### Planner Response

I have identified and resolved the issue causing the database initialization to hang.

### What Was Happening:
When the app initialized, it concurrently requested `Hive.openBox('expenses_box')` and `Hive.openBox('settings_box')`. In our initial implementation:
1. Each box opened its own individual `IndexedDB` connection.
2. The second box requested a database upgrade to create its object store.
3. In IndexedDB, a database upgrade is **blocked indefinitely** if there are other open connections to the same database (namely, the first box's connection) that do not listen to the `versionchange` event to close themselves. This created a silent deadlock, leaving the app permanently stuck on the *"Initializing Hive Databases..."* loading screen.

### How I Fixed It:
1. **Unified Database Manager**: I refactored [src/services/hive.js](file:///D:/Anti%20Gravitiy%20Project/src/services/hive.js) so that all boxes share a single database connection reference managed by `HiveManager` rather than maintaining their own separate instances. 
2. **Pre-creation of object stores**: Both `expenses_box` and `settings_box` are now created together during the initial database creation, avoiding runtime database upgrades entirely. Any future upgrades will safely close existing connections first to prevent blocks.
3. **Troubleshooting View**: I updated [src/App.jsx](file:///D:/Anti%20Gravitiy%20Project/src/App.jsx) with a `dbError` state. If database loading fails (which can happen in strict Private/Incognito tabs where browsers restrict IndexedDB access), the loader will dismiss and render a beautiful troubleshooting page explaining the browser configuration instead of freezing.
